System features

Publishing an email

Description and Priority

A user can forward an email to an email address associated with the app using any email management service (Gmail, Yahoo etc.) and get their email published. Afterwards, they will receive an email containing the link for the published email. The priority for this feature is considered maximum, i.e. 9. Considering the fact that the name of the project itself is Email Publisher we consider this feature as the crucial one.

Stimulus/Response Sequences List

  • A user forwards an email to the dedicated email address of the app, therefore making the view to trigger the controller
    • get the email content and save it in the database
    • generate the link for the newly published email and publish it there
    • send the link for the published email to the user

Functional Requirements

  • an email address which a user can forward emails to and that can send emails to users
  • a database that stores the contents of the published emails and the users

Generating and exporting statistics

Description and Priority

A user can generate the statistics of an email by clicking on the "statistics" button related to a specific email. After that they will be redirected to the "statistics" page where they can find how many views the email has, what are the origin countries of the visitors and the number of visitors per day for a period of a week. The user can modify the grouping category of the visitors diagram and the period for which all the diagrams are shown. Priority of this feature is 5, due to the fact that other features might come up as more needed. Also, this feature is not contained in the core of the project.

Stimulus/Response Sequences List

  • the user is on the dashboard and clicks on the "Statistics" button of a published email
    • the above-mentioned statistics are generated
    • the user is redirected to the "statistics" page associated with the email
  • the user clicks "show statistics for all the emails"
    • the page is refreshed and all the statistics are generated in regards to all the published emails as a whole
  • the user presses the "generate" button after selecting the time period for the statistic
    • all of the statistics are regenerated according to the time period, restricting the time period for which the data is considered
  • the user presses the "generate" button after selecting the time unit for which the diagram of visitors will group data by
    • the visitors diagram will be regenerated and shown to the user in order to group the visitors amount by the selected time unit
  • the user presses an export button from the page
    • a selection of pdf, xml and html file types is given
  • the user selected a format for the statistics to be exported after pressing the "export" button
    • the associated statistics (the statistics above the button) are generated with the specified format and downloaded

Functional Requirements

  • generate statistics for a specific email or all of them in regards to the above-specified constraints
  • export the statistics as pdf, xml or html
  • a database that stores the contents of the published emails and the users

Authentication

Description and Priority

The authentication is needed for associating published emails to users and giving them access to the "Managing published emails" and "Generating and exporting statistics" feature. A user account is created when they forward their first email to our email address or through using our authentication page after submitting a valid token received through email. Due to the fact that authentication is needed for storing the published emails and associating them with specific emails, the priority of this feature is 8.

Stimulus/Response Sequences List

  • user types in an email in the authentication email page and presses the “continue” button
    • send an email to the given email address containing the authentication token which expires in 30 minutes
  • user types in the token received on the mail and presses the “login” button
    • verify if the token isn't expired and that it corresponds to the one sent by email
    • if the token is valid, authentify the user
    • create a new entry in database if the user is new
  • a new user forwards an email to our dedicated email address
    • create a new entry in the users database with the referred email
  • user presses the “go back” button while on the "authentication token" page
    • redirect the user to the authentication email page

Functional Requirements

  • an email address which a user can forward emails to and that can send emails to users
  • a database that stores the users
  • a token generator and validator

Managing published emails

Description and Priority

A user can view all the titles of their published emails on the dashboard. They can also access the associated statistics and access the link that has the published email. Moreover, the user can change the tags of a published email in the dashboard. The tags define: when the published email will be deleted, the absence or presence of a password and what it is, if it is present and if the published email is visible to the user who published it or to everyone. The priority of this feature is 7. This feature is based on the "Publishing an email" and "Authentication" feature. It cannot exist without them, so they have a higher priority.

Stimulus/Response Sequences List

  • the user has authenticated successfully
    • redirect the user to their dashboard
  • change the publication's tags ???
    • change the publication's tags ???
  • clicking the link associated with the publication
    • the user is redirected to the published email's content
  • pressing the “Statistics” button
    • the user is redirected to the published email's statistics

Functional Requirements

  • having all of the other features implemented
  • a database that stores the contents of the published emails and the users

Other non-functional requirements

Security requirements

The project will prevent SQL injection and securely keep the data from being altered or accesed by an attacker.
